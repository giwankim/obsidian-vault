> [!summary]
> Naver's video platform, serving ~60 services, had drifted into two nearly identical UGC and RMC platforms stuck on JDK 6 and Struts 1, doubling every feature's development and deployment cost. In 2017 the team rebuilt it as Video Platform 2.0 on JDK 8 and Spring Boot with an MSA design using Spring Cloud: Eureka for service discovery (with Ribbon, Feign, and direct-client load balancing), Zuul as the API gateway, and Hystrix with per-application thread pools to isolate failures. The article details operational lessons — Eureka's renew/expiration timing defaults, self-preservation mode, and the up-to-90-second traffic drain delay that must be accounted for during zero-downtime deployments.

네이버의 동영상 플랫폼은 2007년에 네이버 블로그에 동영상 서비스를 처음으로 지원했습니다. 현재는 약 60여 개의 네이버 서비스를 지원하고 있으며, 동영상 처리를 위한 API와 도구는 물론 동영상 등록, 변환 처리, 스트리밍, 광고, 플레이어, 데이터 수집 등 여러 범위에 걸쳐 필요한 기능과 도구를 제공합니다. 동영상 플랫폼 담당 부서에서는 각 서비스가 동영상을 쉽게 사용할 수 있도록 기술 지원도 제공합니다.

두 번에 걸쳐 공개할 "동영상 플랫폼 개선기"는 10년 이상 서비스하고 있는 동영상 플랫폼을 새로운 플랫폼으로 개선하며 겪었던 경험을 공유합니다. 기존 동영상 플랫폼과 새로운 동영상 플랫폼의 구성, [Spring Cloud](http://spring.io/projects/spring-cloud) 를 사용해 MSA(micro-service architecture) 기반으로 플랫폼을 변경하는 과정 등을 소개합니다.

첫 번째 글에서는 기존 동영상 플랫폼의 구성과 문제점, 새로운 동영상 플랫폼의 구성을 살펴보겠습니다.

## 동영상 플랫폼의 변화와 개선 배경

네이버의 동영상 플랫폼은 약 60여 개의 네이버 서비스에 동영상 처리를 위한 API와 도구를 제공하는 플랫폼이다.

2007년에 블로그 서비스에 동영상을 처음으로 지원했고, 이후 많은 네이버 서비스에서 동영상이 중요해지면서 새로운 기능을 반영해야 했다. 기능 반영 외에도 동영상 요청량의 증가, 점점 커지는 파일의 크기, 길어지는 재생 시간 등으로 인해 더 많은 시스템 자원이 필요해졌다.

기존 서비스에 영향을 주지 않으면서 새로운 요구 사항을 빠르게 반영하도록 플랫폼을 두 개로 분리하면서 플랫폼의 복잡도도 증가했다. 또한 10년 이상 된 프레임워크를 새로운 프레임워크로 변경하기도 어렵게 됐다. 결국 2017년에 플랫폼의 구성을 다시 설계해 동영상 플랫폼을 개선했다.

### 동영상 플랫폼의 변화

[네이버TV](https://tv.naver.com/) 와 [V LIVE](https://www.vlive.tv/) 와 같이 동영상이 중심인 서비스뿐만 아니라 네이버 서비스 전체에서 동영상이 점차 중요해지면서 동영상 플랫폼에 새로운 기능을 빠르게 적용해 달라는 요구 사항이 많아졌다.

다음은 약 11년 동안 동영상 플랫폼에 반영된 주요 변경 사항이다.

| 변경 사항 | 변경 전 | 변경 후 |
| --- | --- | --- |
| 파일 형식 | FLV | MP4 |
| 지원 해상도 | 단일 해상도 | 멀티 해상도(144p~4K) |
| CDN 연동 방식 | 업로드 방식 | 파일 캐시 방식 |
| 스토리지 | NAS(Network Attached Storage) | - NAS - OwFS(Owner-based File System) |
| 업로드 제약 | - 파일 크기: 100MB 이하 - 재생 시간: 10분 이내 | - 파일 크기: 20GB 이하 - 재생 시간: 16시간 이내 |
| 재생 프로토콜 | [Progressive Download](https://en.wikipedia.org/wiki/Progressive_download) | - Progressive Download - [HLS(HTTP Live Streaming)](https://developer.apple.com/streaming/) - [DASH(Dynamic Adaptive Streaming over HTTP)](https://en.wikipedia.org/wiki/Dynamic_Adaptive_Streaming_over_HTTP) |
| 광고/DRM |  | 광고 동영상 제공, 콘텐츠 보안 제공 |
| 섬네일 | 5장면 섬네일 | - 탐색용 섬네일 - 주요 장면 GIF 섬네일 |
| 해외 지원 |  | 해외에서 동영상 업로드 및 재생 |
| 기타 |  | - 자막 지원 - 360 VR, UHD, 멀티트랙 등의 영상 형식 지원 |

> **OwFS**
> OwFS(Owner-based File System)는 네이버의 대용량 분산 파일 스토리지다. OwFS에 관한 자세한 내용은 " [어떤 분산 파일 시스템을 사용해야 하는가?](https://d2.naver.com/helloworld/258077)"를 참고한다.

이러한 변경 사항 이외에도 동영상 요청량이 증가하고 다루는 파일의 크기가 커질 뿐만 아니라 재생 시간이 더 길어졌다. 대용량 콘텐츠를 처리하기 위해 시스템 내부에서도 많은 변경 작업이 있었고, 플랫폼의 복잡도도 점차 증가했다.

### 두 개로 분리된 플랫폼

2007년부터 서비스를 제공한 동영상 플랫폼은 UGC(user generated contents) 동영상 플랫폼으로, 네이버 블로그, 네이버 카페 등 사용자가 콘텐츠를 등록하는 서비스의 동영상을 처리한다. [네이버TV](https://tv.naver.com/), [V LIVE](https://www.vlive.tv/), 스포츠, 뉴스 등 콘텐츠 공급자가 콘텐츠를 등록하는 서비스가 제공되면서 RMC(ready made contents) 동영상 플랫폼이 필요했다.

RMC 동영상 플랫폼에 요구된 몇몇 기능(멀티 해상도 지원, 스트리밍 프로토콜 추가, 보안 기능 추가)을 기존의 UGC 동영상 플랫폼에 영향을 주지 않고 짧은 일정 안에 제공할 수 있는 가장 간단한 방법은 UGC 동영상 플랫폼 코드를 가져와 새로운 플랫폼을 구축하는 것이었다. 그래서 2011년에 UGC 동영상 플랫폼과 약간은 다르지만 기능은 거의 동일한 RMC 동영상 플랫폼을 별도의 IDC에 구축했다.

하지만 두 개의 플랫폼을 서로 다른 담당자가 개발하고 운영하면서 시간이 지날수록 개발과 운영에 모두 부담이 생겼다. 새롭게 요구되는 기능에는 UGC 동영상 플랫폼과 RMC 동영상 플랫폼에 종속된 기능도 있었지만 대부분의 기능은 양쪽 플랫폼 모두에 필요한 기능이었다. 그래서 새로운 기능을 제공할 때마다 오랫동안 달라진 코드에 맞추어 중복으로 개발해야 했고 배포도 별도로 해야 했다.

### 오래된 프레임워크

기존 동영상 플랫폼의 애플리케이션 대부분은 10년 전에 선택된 JDK 6과 Struts 1을 기반으로 개발되어 있었다. 또한 XML 형태의 응답값을 JSP에 직접 코딩하는 방식과 같이 지금은 API 서버에서 잘 사용하지 않는 패턴을 사용했다.

최근에 나온 일부 오픈소스 라이브러리는 더 이상 JDK 6을 지원하지 않는다. 이는 편리하고 강력한 기능을 제공하는 최신 프레임워크를 사용하기 어렵다는 의미이다. 이처럼 오래된 기술을 계속 유지 보수하면 생산성이 저하될 뿐만 아니라 개발자의 의욕도 저하될 수 있다. 변화가 필요했다.

## 동영상 플랫폼의 개선 방향

플랫폼별로 담당자를 분리해 개발하고 운영하는 것이 아니라 주요 기능을 담당하는 애플리케이션별로 담당자를 지정하도록 변경할 필요가 있었다. 그리고 플랫폼에 따라 두 개로 분리된 각 구성 요소를 하나로 통합할 필요도 있었다.

다음과 같은 방향으로 동영상 플랫폼을 개선하기로 했다.

- 두 개로 나누어진 플랫폼을 하나로 통합한다.
- 기존의 플랫폼과 호환되도록 한다.
- 각 모듈이 독립적으로 작동하고 확장될 수 있도록 구성한다.

## 기존 동영상 플랫폼의 주요 구성

기존의 동영상 플랫폼을 살펴보면 다음과 같이 구성되었다. 다른 부분도 있지만 UGC 동영상 플랫폼과 RMC 동영상 플랫폼의 주요 구성은 동일하다.

![그림 1](https://d2.naver.com/content/images/2019/04/helloworld-201904-vdoplatform_1-1.png)

그림 1 기존 동영상 플랫폼의 구성

각 구성 요소의 기능은 다음과 같다.

- Uploader: 동영상 파일을 업로드한다. 동영상 파일의 헤더와 섬네일을 추출한다. 서비스별 트랜스코딩 설정에 따라 동영상 메타 정보를 생성한다.
- Encoder: 트랜스코딩을 처리한다.
- VideoUtility: HLS(HTTP Live Streaming)를 위해 하나의 동영상을 패킷으로 나누는 작업을 처리한다.
- ServiceAPI: 동영상의 메타 정보를 관리한다. 영상의 등록부터 재생까지 모든 API를 제공한다.
- Object Storage: 동영상 파일을 저장한다. NAS에 동영상의 원본 파일을 임시로 보관하고, OwFS에 동영상의 결과물 파일을 저장한다.
- Cache: 데이터베이스 조회 결과나 API 응답 등을 분산 메모리 캐시 클러스터(Arcus)에 저장해 데이터베이스 부하와 API의 지연 시간(latency)을 줄인다.
- Database: 동영상의 메타 정보를 MySQL에 데이터베이스로 저장한다.
- CDN Origin: CDN의 원본 서버 역할을 하며, 동영상 파일을 제공한다.

> **Arcus**
> Arcus는 네이버의 memcached 기반 분산 메모리 캐시 클라우드다. Arcus에 관한 자세한 내용은 GitHub의 [Arcus 프로젝트 사이트](https://github.com/naver/arcus) 를 참고한다.

## 새로운 동영상 플랫폼의 주요 구성

UGC 동영상 플랫폼과 RMC 동영상 플랫폼을 통합할 수 있는 새로운 동영상 플랫폼이 필요했지만, 이 작업은 단기간에 진행할 수 없는 작업이다. 그래서 우선 확장성을 고려하면서 기존 두 플랫폼 중 RMC 동영상 플랫폼을 대체하는 플랫폼을 만들기로 했다.

다음은 새로운 동영상 플랫폼인 동영상 플랫폼 2.0의 구조이다.

![그림 2](https://d2.naver.com/content/images/2019/04/0a705587-6a10-1a4c-816a-1ed7ffba30af.png)

그림 2 동영상 플랫폼 2.0의 구성

운영하던 애플리케이션의 대부분은 웹 페이지 제공보다 API 제공을 더 많이 하는 애플리케이션이었기 때문에 [Spring Boot](http://spring.io/projects/spring-boot) 를 기반으로 애플리케이션을 구성했다. 또한 플랫폼을 개선하기 시작한 2017년 1월에 최신 프레임워크였던 JDK 8을 기반으로 구성했다.

### MSA 기반 구성

동영상 플랫폼 2.0에서는 각 애플리케이션 사이의 의존성을 낮추고 독립적으로 빠르게 애플리케이션을 개발하고 확장하기를 원했다. 기존에는 하나의 애플리케이션이 여러 역할을 담당하기도 했고, 일부 모듈은 구조적인 문제로 인해 배포 시간이 매우 오래 걸리기도 했다. 이 부분을 해결하기 위해서 각 역할에 맞게 애플리케이션을 분리했다.

동영상 애플리케이션은 특성에 따라 CPU나 I/O를 매우 많이 사용한다. 또한 동영상 플랫폼은 수천 대 이상의 장비로 구성되어 있다. 그래서 동영상 플랫폼에서는 리소스를 효율적으로 사용하는 것이 중요하다. 장기적인 관점에서 MSA 기반 구조와 함께 컨테이너 기반의 운영도 염두에 두었다.

MSA 기반으로 동영상 플랫폼을 구성할 때 다음 두 가지 사항이 주요 고민이었다.

- MSA 구성으로 인한 시스템 전체의 복잡도를 어떻게 낮출 수 있을까?
- 기존 서비스에 영향을 미치지 않고 어떻게 안정적으로 MSA 구성으로 전환할 수 있을까?

동영상 플랫폼 2.0을 설계하던 때에는 [Netflix OSS](https://netflix.github.io/) 를 기반으로 한 Spring Cloud가 자리를 잡아가고 있었다. 이미 다음과 같은 구현체가 있어 MSA의 주요 컴포넌트인 서비스 디스커버리(service discovery)와 API 게이트웨이, 서킷 브레이커(circuit breaker)의 구현이 큰 부담이 되지 않았다.

- [Eureka](https://github.com/Netflix/eureka): 서비스 디스커버리의 구현체
- [Zuul](https://github.com/Netflix/zuul): API 게이트웨이의 구현체
- [Hystrix](https://github.com/Netflix/Hystrix): 서킷 브레이커의 구현체

또한 설정을 도와주는 [Spring Cloud Config](https://cloud.spring.io/spring-cloud-config/), 모니터링을 위한 [Spring Cloud Sleuth](https://spring.io/projects/spring-cloud-sleuth) 와 [Zipkin](https://github.com/openzipkin/zipkin) 등의 도구도 Spring Boot와 쉽게 연동할 수 있었다.

여러 MSA 컴포넌트 중 우선 동영상 플랫폼 2.0 구성에 필요한 서비스 디스커버리와 API 게이트웨이를 구축했다.

### Eureka를 사용한 서비스 디스커버리 구축

기존에는 대부분 L4를 기반으로 로드 밸런싱을 했다. MSA는 monolith 방식 아키텍처보다 모듈 수가 매우 많다. 애플리케이션은 통신을 통해 서로 데이터를 주고받거나 기능을 수행한다. 많은 수의 애플리케이션을 기존처럼 L4 방식으로 운영하면 운영 비용이 모듈 수만큼 증가한다.

또한 분산 환경에서는 배포 전략에 따라 매번 네트워크 주소가 변경될 수 있다. 사내 인프라 업무 요청 시스템을 통해 인프라 업무를 진행하는 네이버에서는 L4 등의 네트워크 작업을 처리할 때는 장애 상황이 아니라면 처리에 3일 정도가 소요된다. L4 방식으로는 MSA 기반 구성을 원활하게 운영할 수 없다.

그래서 Eureka를 사용해 서비스 디스커버리를 적용했다. 서비스 디스커버리는 매번 네트워크 주소가 변경될 수 있는 분산 환경에서 노드 사이의 통신을 위해 네트워크 주소를 찾는 패턴이다. Eureka는 Netflix OSS가 제공하는 서비스 디스커버리의 구현체다.

#### 서비스 디스커버리와 로드 밸런싱

다음은 Eureka의 아키텍처를 나타낸 그림이다.

![그림 3](https://d2.naver.com/content/images/2019/04/helloworld-201904-vdoplatform_1-3.png)

그림 3 Eureka의 아키텍처(원본 출처: [Eureka at a glance](https://github.com/Netflix/eureka/wiki/Eureka-at-a-glance))

애플리케이션 서비스(Application Service)는 Eureka 클라이언트로, 자신의 네트워크 주소 정보와 상태를 주기적으로 Eureka 서버에 등록하고 갱신한다. Eureka에서는 이렇게 모아진 애플리케이션의 정보를 'Registry 정보'라고 부른다. 역시 Eureka 클라이언트인 애플리케이션 클라이언트(Application Client)는 다른 애플리케이션 서비스와 통신하기 위해 Eureka 서버에서 받은 Registry 정보를 활용한다.

애플리케이션 서비스는 일반적으로 여러 대의 장비로 구성되며, 애플리케이션 클라이언트가 받은 Registry 정보에는 여러 네트워크 주소가 포함되어 있다. 그렇기 때문에 애플리케이션 클라이언트가 받은 여러 네트워크 주소 중에서 어느 주소로 요청을 보내야 하는지 선택하는 로드 밸런싱을 구현해야 한다.

Spring Cloud는 Eureka 기반 로드 밸런싱을 쉽게 사용할 수 있게 [Ribbon 방식 구현체(RibbonClient)](https://cloud.spring.io/spring-cloud-netflix/multi/multi_spring-cloud-ribbon.html) 와 [Feign 방식 구현체(FeignClient)](https://cloud.spring.io/spring-cloud-netflix/multi/multi_spring-cloud-feign.html) 를 지원한다. 구현체는 애너테이션으로 선언해 사용할 수 있다.

동영상 플랫폼 2.0에서는 다음과 같은 3가지 방식을 사용해 로드 밸런싱을 구현했다.

- Ribbon 방식: Eureka와 연동해 간단히 로드 밸런싱을 사용할 때나 호출해야 하는 API가 명시적이지 않을 때(호출 URL을 파라미터로 주입받는 경우) 사용한다.
	Spring 환경 설정에서 @EnableDiscoveryClient 애너테이션을 선언하고 다음 예와 같이 RestTemplate Bean에 @LoadBalanced 애너테이션을 선언한다. 나머지는 RestTemplate Bean을 사용하는 방법과 동일하다.
```java
@LoadBalanced
@Bean
public RestTemplate loadBalanced(RestTemplateBuilder restTemplateBuilder) {
    return restTemplateBuilder
            .messageConverters(this.getMessageConverters())
            .setConnectTimeout(2000)
            .setReadTimeout(2000)
            .errorHandler(new LoadBalancedErrorHandler())
            .build();
}
```
- Feign 방식: 내부 통신 등 API가 명확한 경우나 Ribbon 방식의 로드 밸런싱에 서킷 브레이커(Hystrix)를 적용하고 싶을 때 사용한다. 다음 예와 같이 API를 메서드 형태로 명시적으로 사용할 수 있다.
	Spring 환경 설정에서 @EnableFeignClients 애너테이션을 선언해야 한다.
```java
@FeignClient("vodapi")
public interface VideoFeignClient {

    @GetMapping(value = "/videos/v1/{videoId}", produces = MediaType.APPLICATION_JSON_VALUE)
    Video getVideo(
            @RequestHeader(Constants.KEY) String key,
            @PathVariable("videoId") String videoId,
            @RequestParam("webPath") Boolean webPath
    );

}
```
- Eureka 클라이언트를 통해 직접 로드 밸런싱 구현: 애플리케이션 클라이언트의 Registry 정보에서 특정 조건에 해당하는 서버의 목록을 받을 때 사용한다.
	다음 예와 같이 Eureka에 등록된 애플리케이션 이름으로 애플리케이션 클라이언트의 정보를 얻을 수 있다.
```java
@Autowired
private EurekaClient discoveryClient;

private List<String> getAvailableAddresses(Zone zone) {
    return discoveryClient.getApplication(applicationName).getInstances().stream()
            .filter({조건})
            .collect(Collectors.toList());
}
```

Eureka를 도입해 로드 밸런싱을 구현할 때에는 다음과 같은 점을 고려해야 한다.

- Eureka 설정값에 따른 내부 동작 이해
- 장애 패턴에서의 주의할 점
- 배포 전략(트래픽 제어)

#### Eureka 설정값에 따른 내부 동작 이해

대부분의 Eureka 설정에는 기본값이 지정되어 있어서 서버를 구축하거나 Eureka 클라이언트를 사용할 때에 크게 어려움이 없다. 하지만 운영 환경에서 안정적으로 Eureka를 사용하려면 내부 동작과 기본값에 대해 이해해야 한다.

Eureka 인스턴스는 leaseRenewalIntervalInSeconds 설정(기본값: 30초)에 의해 주기적으로 Eureka 서버에 자신의 상태를 알려준다(renew). Eureka 인스턴스의 leaseExpirationDurationInSeconds 설정에 지정된 시간(기본값: 90초) 동안 Eureka 인스턴스에서 renew 요청이 오지 않으면 Eureka 서버는 해당 Eureka 인스턴스가 작동하지 않는다고 판단하고 이를 제외한다.

Eureka 클라이언트는 registryFetchIntervalSeconds 설정에 지정된 시간(기본값: 30초)마다 서버에서 Registry 정보를 받아와서 로컬 캐시에 저장해 사용한다. Registry 정보의 변경 사항은 instanceInfoReplicationIntervalSeconds 설정(기본값: 30초)에 의해 주기적으로 복제된다(replication). Eureka 서버는 애플리케이션 시작 시 다른 Eureka 서버와 Registry 정보를 동기화한다. 그 외의 동작에서는 Eureka 클라이언트의 설정과 작동 방식을 따른다.

#### 장애 패턴에서 주의할 점

Eureka 서버는 enableSelfPreservation 설정을 통해 [self preservation 모드](https://github.com/Netflix/eureka/wiki/Server-Self-Preservation-Mode) 를 제어할 수 있다(기본값: `true`). 이 설정은 네트워크 문제로 Eureka 클라이언트가 대규모로 누락되는 사태를 방지하기 위한 설정이다. Eureka 서버가 Eureka 클라이언트에서 받아야 하는 renew 요청이 특정 임곗값(기본값: 15분 내 받아야 하는 renew 요청에 대한 기댓값의 85%) 이하가 되면 self preservation 모드가 작동한다. 이때 Eureka 클라이언트에서 더 이상 renew 요청이 없어도 Registry 정보에서 해당 클라이언트를 제외하지 않고 보존한다.

Eureka를 구축해 운영할 때에는 최소한 이러한 내용을 인지하고 있어야 예상하지 못한 동작을 막을 수 있다.

#### Eureka 사용 시 배포를 위한 트래픽 제어

Eureka의 작동 방식이나 설정은 배포 전략에도 고려되어야 한다. 네이버의 동영상 플랫폼은 자체 인프라에서 인스턴스를 고정해서 사용하고 일부 서버의 트래픽을 제어해서 무중단 배포를 진행하는 전략을 사용한다.

Eureka에서는 InstanceStatus 객체의 상태(UP, DOWN, STARTING, OUT *OF* SERVICE, UNKNOWN)에 따라 해당 Eureka 인스턴스로 트래픽을 보낼지 말지가 결정된다. Eureka 적용 전에는 배포 시 L7 Health Check를 사용해 트래픽을 제어했다. 따라서 기존 배포 방식과 인터페이스를 그대로 사용하기 위해 다음과 같은 작업을 했다.

- 초기 상태를 DOWN 상태로 만들기 위해 Eureka의 `initial-status` 설정을 `DOWN` 으로 설정한다.
- healthcheck.enabled 설정을 `true` 로 설정한다.
- 다음과 같이 VodHealthCheckHandler 클래스를 구현해 기존 인터페이스에서 사용한다.
```java
public class VodHealthCheckHandler implements HealthCheckHandler {
   private final VodHealthIndicator vodHealthIndicator;

   public VodHealthCheckHandler(VodHealthIndicator vodHealthIndicator) {
      this.vodHealthIndicator = vodHealthIndicator;
   }

   @Override
   public InstanceStatus getStatus(InstanceStatus currentStatus) {
      log.info("eureka instance status : {} -> {}", currentStatus, this.vodHealthIndicator.getStatus());

      if (!Status.UP.equals(this.vodHealthIndicator.getStatus())) {
         currentStatus = InstanceStatus.DOWN;
      } else {
         currentStatus = InstanceStatus.UP;
      }

      return currentStatus;
   }
}
```

배포 시 위와 같이 트래픽을 제어할 때 주의해야 할 점은 Eureka의 설정에 영향을 받는다는 점이다. 기본 설정을 사용한다면 Eureka 인스턴스 제외를 명령한 서버로 실제로 트래픽이 전달되지 않을 때까지 최대 90초가량이 소요될 수 있다. Eureka 인스턴스의 상태가 변경되고, 이 정보를 Eureka 서버에 renew를 통해 알리고, Eureka 서버 간 복제 및 Eureka 클라이언트가 Eureka 서버로부터 Registry 정보 갱신을 완료해야 Eureka 클라이언트에서 로드 밸런싱 시 더 이상 해당 Eureka 인스턴스로 트래픽을 보내지 않는다. 따라서 배포 시에는 트래픽에 대한 확인이 필요하다.

### Zuul을 사용한 API 게이트웨이 구축

Eureka를 통해 내부 애플리케이션이 서로 통신하는 방법에 대해 설명했다. 내부 통신뿐만 아니라 외부에서 들어오는 트래픽에 대한 부분도 고민해야 했다. 모든 클라이언트에서 로드 밸런싱을 구현하기는 불가능에 가까우므로, API 진입점은 하나로 유지하고 외부로부터 들어오는 요청을 각 애플리케이션으로 라우팅할 API 게이트웨이가 필요했다.

![그림 4](https://d2.naver.com/content/images/2019/04/helloworld-201904-vdoplatform_1-4.png)

그림 4 해외 지원 환경에서 API 통신

해외 지원을 위해 API 게이트웨이는 한국뿐 아니라 여러 POP(point of presence)에 배치했다. GSLB(global server load balancing) 방식을 통해 외부의 요청은 가장 가까운 곳에 있는 API 게이트웨이로 전달되고, API 게이트웨이를 통해 백엔드에 있는 각 서비스로 라우팅된다.

API 게이트웨이에는 동적으로 라우팅을 처리하기 위해 Spring Cloud의 API 게이트웨이 구현체인 Zuul이 탑재되어 있다. 백엔드에 있는 서비스 애플리케이션은 주기적으로 Eureka에 Registry 정보를 등록하고, Zuul은 이 정보를 바탕으로 외부의 요청을 내부 서비스로 라우팅한다. 이런 동적 라우팅은 장비의 추가, 삭제 시나 서비스 추가 시 별도의 운영 작업이 없어도 신규 API를 동적으로 등록해 사용할 수 있게 한다.

#### API 게이트웨이 구성

API 게이트웨이는 그림 4에서와 같이 백엔드에 있는 여러 서비스로 요청을 라우팅해야 한다. 요청을 알맞은 백엔드 서비스로 라우팅하기 위해 백엔드에 어떤 서비스가 있는지, 서비스는 어떤 주소로 구성되어 있는지, 어떻게 로드 밸런싱을 해야 하는지 등을 API 게이트웨이는 알고 있어야 한다. 또한 발생 가능한 여러 장애 상황에서도 안정적으로 운영할 수 있는 기능이 필요하다. Zuul은 API 게이트웨이 구성에 필요한 기능을 Spring Cloud의 다른 컴포넌트인 Eureka와 Ribbon, Hystrix와 함께 구성해 해결할 수 있다.

API 게이트웨이를 구성할 수 있는 컴포넌트와 역할은 다음과 같다.

- Zuul: 라우터와 필터 기능을 수행한다.
	- 사용자 설정 혹은 Eureka 설정에서 지정한 라우팅 정책에 따라 요청을 라우팅한다.
		- 필터를 통해 요청에 대한 인증을 처리한다.
		- 필터로 요청에 대한 지표나 데이터를 수집한다.
- Eureka: 서비스 디스커버리 기능을 수행한다.
	- Zuul은 Eureka 클라이언트가 되며, 요청을 라우팅할 각 애플리케이션의 Registry 정보를 받아온다.
- Ribbon: 클라이언트 로드 밸런싱을 담당한다.
	- Eureka에서 받은 Registry 정보를 바탕으로 클라이언트 로드 밸런싱과 커넥션 관리를 수행한다.
- Hystrix: 서킷 브레이커 기능을 수행한다.
	- 특정 서비스의 장애로 인해 API 게이트웨이 전체에 장애가 전파되지 않게 서킷을 관리한다.
		- 특정 서비스의 요청량 증가로 인해 다른 서비스가 영향을 받지 않도록 서비스별 스레드 풀(thread pool)을 두어 서비스를 격리한다.

#### 안정적인 게이트웨이 운영을 위한 고려 사항

Eureka와 마찬가지로 Zuul 또한 운영 환경에서 사용 시 주의를 요한다. API 게이트웨이는 모든 API의 진입점이 되므로 게이트웨이 장애는 전체 장애를 의미한다. 그래서 API 게이트웨이에서는 안정성이 매우 중요하다.

백엔드에 있는 일부 애플리케이션에 장애가 발생하거나 응답 속도가 늦어지면 스레드 풀이 고갈되어 전체 장애로 이어질 수 있다. 이런 장애를 막기 위해 의존성이 있는 애플리케이션마다 별도의 스레드 풀을 두어 격리할 필요가 있다.

또한 API별로 타임아웃을 적절히 설정해야 한다. 하지만 각 API마다 요청량도 다르고 응답 속도 또한 달라서 상황에 따라 타임아웃 값이 변하기 때문에 적절한 타임아웃을 설정하는 것이 어렵다.

#### 장애 전파 방지를 위한 Hystrix 설정

MSA 환경에서는 애플리케이션이 분리되고 많아지면서 기존에는 애플리케이션 하나에서 처리 가능한 부분도 여러 번의 API 통신이 필요하다. 다음 그림처럼 애플리케이션이 다른 애플리케이션에 의존 관계를 가지고 있을 때 한 쪽의 장애로 인해 장애가 전파될 수 있다.

![그림 5](https://d2.naver.com/content/images/2019/04/helloworld-201904-vdoplatform_1-5.png)

그림 5 의존 관계에 있는 애플리케이션의 장애 전파

분산 환경에서 장애가 전파되는 것을 막고 안정적으로 서비스하기 위해 다음과 같은 Resiliency 원칙을 기반으로 여러 패턴을 모아 놓은 구현체가 Hystrix다.

- 빠른 실패 처리와 복구(fail fast & rapidly recover)
	- 요청이 너무 많거나 문제가 발생해 더 이상 요청을 처리할 수 없다면 빠르게 실패로 처리한다.
		- 장애가 지속되는 것을 방지하기 위해 문제가 발생한 곳으로 트래픽을 더 이상 보내지 않는다.
- 실패 대체(fallback)
	- 가능하다면 실패를 대체할 수 있는 처리를 해 응답한다.
- 격리 처리(bulkhead)
	- 하나에서 문제가 발생해도 나머지는 정상으로 작동하도록 스레드 풀을 두어 격리한다.
- 서킷 브레이커
	- 실패율을 모니터링해 문제가 발생하면 처리 요청을 차단한다. 정상으로 회복되면 다시 요청을 처리한다.
		- 서킷이 Closed 상태이면 문제가 없는 상태다. 서킷의 초기 상태다.
		- 서킷이 Open 상태이면 실패율이 임곗값을 넘어간 상태다. 더 이상 요청을 처리하지 않고 빠르게 실패로 처리한다.
		- 서킷이 Half-Open 상태이면 Open 상태에서 일정 시간이 지난 상태다. 요청을 시도해 성공하면 Closed 상태로, 실패하면 Open 상태로 돌아간다.

API 게이트웨이는 요청을 라우팅하는 모든 백엔드 애플리케이션에 의존 관계를 가지고 있다. 특정 애플리케이션의 장애가 API 게이트웨이 전체에 영향을 주는 것을 방지하려면 API 게이트웨이 설정 시 각 애플리케이션마다 별도의 Hystrix 스레드 풀을 두어 애플리케이션을 격리해야 한다. 또한 애플리케이션별로 요청량(rps)과 타임아웃 등을 고려해 스레드 풀의 개수를 지정해야 한다.

Hystrix는 다양한 옵션을 제공해 여러 환경에 맞게 옵션을 사용할 수 있다. Hystrix의 옵션에 관한 자세한 내용은 [GitHub 저장소의 Wiki 문서](https://github.com/Netflix/Hystrix/wiki) 를 참고한다.

#### Zuul 설정 예

다음은 동영상 플랫폼 2.0에 적용되어 있는 API 게이트웨이 설정의 일부다(2018년 11월 기준).

```java
server.tomcat.max-threads: {tomcat max thread 수}

zuul:
  retryable: true
  ignoredServices: '*'   // 지정하지 않은 서비스는 무시
  routes:
    A: /A/**    // A 서비스 라우팅
    B: /B/**    // B 서비스 라우팅
  ribbon-isolation-strategy: THREAD    // isolation 전략 THREAD 권장
  threadPool:
    useSeparateThreadPools: true    // (주의) 각 애플리케이션 별로 ThreadPool 분리해 사용
    threadPoolKeyPrefix: gw        // Thread Pool의 prefix

ribbon:
  MaxTotalConnections: {최대 connection 수}
  MaxConnectionsPerHost: {호스트 당 최대 커넥션 수}
  ReadTimeout: {기본 connection timeout}
  ConnectTimeout: {기본 connection timeout}

A.ribbon.ReadTimeout: 250
A.ribbon.ConnectTimeout: 300

B.ribbon.ReadTimeout: 1500
B.ribbon.ConnectTimeout: 300

hystrix:
  threadpool:
    gwA:
      coreSize: {A의 Thread Pool 사이즈}
      maximumSize: {A의 Thread Pool max 사이즈}
    gwB:
      coreSize: {B의 Thread Pool 사이즈}
      maximumSize: {B의 Thread Pool max 사이즈}
  command:
    A:
      execution:
        isolation:
          strategy: THREAD
          thread.timeoutInMilliseconds: 350
    B:
      execution:
        isolation:
          strategy: THREAD
          thread.timeoutInMilliseconds: 1600
```

API 게이트웨이를 통해 들어온 요청 A와 B를 백엔드에 있는 애플리케이션으로 라우팅한다. 격리 전략에는 SEMAPHORE 방식과 THREAD 방식이 있다. 요청량이 많거나 장애가 발생하지 않는 안정적인 환경을 제외하고는 THREAD 방식을 사용하는 것이 권장된다. 동영상 플랫폼 2.0의 API 게이트웨이에서는 THREAD 방식을 사용했다. 애플리케이션으로 연결하는 HTTP 통신에 대한 커넥션 설정과 타임아웃 설정은 Ribbon 설정으로 제어할 수 있다. Hystrix 설정에서는 A 애플리케이션과 B 애플리케이션에 대한 스레드풀과 스레드 타임아웃을 설정할 수 있다.

다음 그림은 Hystrix의 스레드 풀을 지정하는 방법에 대한 기본 가이드이다.

스레스 풀과 타임아웃은 애플리케이션의 요청량과 응답 시간 등을 고려해 설정한다. 하지만 게이트웨이에서는 상황이 조금 다르다. 게이트웨이에서 설정 가능한 격리 단위는 API가 아니라 애플리케이션이다. 어떤 애플리케이션의 경우에는 API마다 지연 시간이 다르고 비중 또한 다르다. 이런 경우에는 읽기 타임아웃(read timeout)을 응답속도가 느린 API를 기준으로 설정할 수밖에 없다. 스레드 타임아웃을 설정할 때에는 문제 발생 시 스레드가 빠르게 고갈되는 것을 방지하기 위해 재시도를 위한 시간 간격을 짧게 설정할 수 밖에 없다.

## 마치며

네이버 동영상 플랫폼의 개선 배경과 방향, 그리고 신규 플랫폼을 MSA로 구성하면서 고민했던 점을 알아보았다.

UGC 동영상 플랫폼과 RMC 동영상 플랫폼으로 분리되어 있던 기존의 동영상 플랫폼은 개발과 운영에 모두 부담을 주었다. 그리고 JDK 6과 Struts 1을 기반으로 하는 프레임워크는 생산성의 저하를 가져왔다. 두 동영상 플랫폼의 통합과 확장성을 고려하면서 우선 RMC 동영상 플랫폼을 JDK 8을 기반으로 새로 구성했다. MSA 기반으로 새로 구성한 동영상 플랫폼 2.0에서는 Eureka와 Zuul을 사용해 내부 트래픽과 외부 트래픽을 처리하게 했다.

새로운 동영상 플랫폼을 구축하면서 아쉬웠던 점은 오토 스케일링에 대한 부분이었다. 장비를 트래픽에 따라 동적으로 늘릴 수 없기 때문에 용량에 여유를 많이 두어야 했다. 이 부분은 추후 컨테이너 클러스터로 플랫폼을 옮기면서 개선할 예정이다.

다음 글에서는 새로운 동영상 플랫폼의 각 구성 요소를 개발하면서 고민했던 부분, 기존 플랫폼과 하위 호환성을 유지하며 새로운 동영상 플랫폼을 구축한 과정, 동영상 플랫폼을 하나로 통합하면서 겪었던 경험 등을 설명하겠다.
